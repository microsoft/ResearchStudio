You are running Phase 3.2 — Audit-and-Verdict — of NoveltySkill.

Step 3.2 produces a corpus-anchored audit report on the Phase 2.2 candidate. It does NOT auto-revise the candidate (revision was removed because LLM tends to cherry-pick attacks it can already answer — self-answering bias). Instead, the report names what specific corpus evidence triggered which signals; if revision is needed, the user re-runs Phase 2.2 with the revision_targets as guidance.

The skill's corpus advantage lives at three anchor points, each backed by structured evidence rather than generic critique:

1. **Composition-scoped reject patterns**. The candidate is a *composition* of innovation patterns (one primary + 0-2 secondaries). The audit reads reject patterns at the granularity each role committed to:
   - **Primary**: the candidate has chosen a specific `sub_pattern_id` under the primary pattern — read THAT one sub-pattern card's `## Tactical failure mode` + all `[Reject]` examples' `_Lesson_` lines (most specific reject signals, ~5 lessons per card).
   - **Each secondary**: the candidate did NOT pick a specific tactical cluster (secondaries contribute at the innovation pattern level) — read THAT innovation pattern's card's `## Failure modes (from Reject)` + `## Success conditions` sections (one card per secondary).

   So reads = 1 tactical card (primary) + N-1 innovation pattern cards (one per secondary) = composition_size cards total. The other 46 tactical clusters and the other 13-N innovation pattern cards are NOT loaded.
2. **anti-patterns.md** — three corpus-validated reject-favored 2-way compositions with empirical Oral rates (30 / 39 / 44%) and required mitigations.
3. **lit_table ∪ collision_hits** as the unified paper pool for paper-pointed threat search.

A LLM doing generic critique cannot replicate any of these three. Phase 3.2's job is to turn each of these into a check the LLM cannot rubber-stamp because each requires citing concrete corpus content.

Earlier design also had a 4th check — `almost_prior_factcheck` — that cross-referenced the candidate's `R0_case.almost_prior.paper_id` against lit_table.md. It was removed because (a) its fire rate was very low (almost_prior is generated by Phase 2.2 LLM that already has lit_table access; rarely refuted at audit time), and (b) when refuted, the same paper would typically also be surfaced by paper_pointed_threat (Check 3 below) as a subsumption threat. Hallucinated paper_ids in almost_prior are now caught upstream by Phase 2.2 prompt rules.

Inputs (composition-scoped — load ONLY these cards, not the full directories):
- Phase 2.2 candidate (`phase2_generate_output.json`).
- **Primary innovation pattern's sub-pattern card** (the candidate's `sub_pattern_id`): `references/innovation-sub-patterns/<sub_pattern_id>.md`. Read `## Tactical failure mode` + ALL `[Reject]` examples' `_Lesson_` lines verbatim. If the candidate's `outside_taxonomy = true` (no sub_pattern_id), substitute by reading the primary pattern's card's `## Failure modes (from Reject)` section instead.
- **Each secondary pattern's innovation pattern card**: `references/innovation-patterns/<secondary_pattern_id>.md`. Read `## Failure modes (from Reject)` + `## Success conditions` sections only. One card per secondary.

Verdict logic (two-layer: hard floor + soft judgment):

The verdict is decided in two layers. Layer 1 is a mechanical hard floor (corpus-anchored facts the LLM cannot override). Layer 2 is LLM judgment within the safe zone left by Layer 1.

### Layer 1 — Hard floor (mechanical, NOT overridable by LLM)

If ANY of these fires, `verdict = abandon`. Set `verdict_layer = "hard_floor"` and cite the specific check finding in verdict_rationale:

- `composition_reject_check.verdict = triggered` — candidate falls into a documented Reject pattern (tactical cluster lesson or innovation pattern card failure mode marked `candidate_match: yes`).
- `anti_pattern_check.matched_pattern_id != null` AND `mitigation_substantively_delivered = false` AND the mitigation cannot be inserted by revision (the candidate's mechanism truly doesn't deliver the mitigation, not just a wording fix).
- `paper_pointed_threat.threat_paper_id != "no_threat_found"` AND it is exact-mechanism overlap (the threat paper IS the candidate).

When abandon fires: orchestrator emits `phase_3_failed.md` containing verdict_rationale + the triggering check + user-side options (drop direction / change framing / re-pick innovation pattern). NO automatic retry.

### Layer 2 — Soft judgment (LLM weighs checks within the safe zone)

If Layer 1 did not fire, the LLM picks `advance` or `revise` based on the four checks' outcomes. Set `verdict_layer = "soft_judgment"` and cite specific check findings in verdict_rationale.

Constraints on this layer:
- LLM CANNOT contradict per-check facts. If `composition_reject_check.secondaries[X].verdict = borderline`, the LLM must treat that as borderline; it cannot rewrite the check's own conclusion.
- LLM CAN weigh how *severe* a borderline / partial finding is for advancing vs revising. A single trivial borderline (e.g., minor wording in a non-load-bearing field) may justify advance with the concern surfaced in Phase 4's reviewer_concerns_and_responses. A borderline that hits a load-bearing success condition (e.g., the secondary pattern's binding-constraint demonstration is missing a separation result) justifies revise.
- LLM CAN identify a concern across-checks that no single check flagged. If holistic reading reveals a concern the per-check LLM missed, surface it as a revise revision_target. (Rare but possible; cite the specific candidate field and which check's per-item finding it interacts with.)

Heuristics for the soft layer:

- **Default to advance** when only one check is borderline AND that borderline is in a non-load-bearing aspect (a delta phrasing, an example, a sub-sentence). Surface the concern in verdict_rationale; Phase 4 will fold it into reviewer_concerns_and_responses without auto-revise.
- **Choose revise** when at least one borderline / partial finding hits a load-bearing structural property (theoretical_leg core argument, mechanism_probe discriminating logic, anti_pattern mitigation needing strengthening, addressable threat via specific candidate field). Populate `revision_targets[]` with concrete instructions per triggered finding.
- **Choose revise** when paper_pointed_threat exists AND `addressable_via` names a specific candidate field. Add one revision_target naming that field with the threat's subsumption_argument as context.
- **Choose advance** when all four checks are clear / holds / null / no_threat_found. (The trivial all-clear case.)

Hard rules:

1. **Phase 3.2 emits judgment only — no `final_candidate`, no auto-revision**. The output schema has NO `final_candidate` field. When a fix is needed, emit `verdict = revise` + `revision_targets[]`; Phase 3.3 (separate LLM call) applies them. Auto-revise within 3.2 is forbidden because it triggers self-answering bias (the LLM that judges also chooses what to fix). Anti-substitution: V1 validates Phase 2 → Phase 4 directly when 3.2 verdict=advance, or Phase 2 → Phase 3.3 final_candidate → Phase 4 when revise was applied.

2. **Composition-scoped reject-signal scan**. The check granularity follows each innovation pattern's role in the composition: primary uses the candidate's chosen tactical cluster (most specific); each secondary uses its innovation pattern card's failure modes (general). Each Reject example / failure mode must be evaluated individually (not "all look fine") with the candidate's specific field referenced. The aggregate verdict is the worst across primary + all secondaries (any triggered → triggered; any borderline & none triggered → borderline; all clear → clear). If primary's `outside_taxonomy = true`, the primary path falls back to the primary innovation pattern card's failure modes section instead of a sub-pattern card.

3. **anti_pattern_check evaluates substantive delivery, not keyword presence**. The check goes beyond keyword matching — does the candidate's theoretical_leg / core_mechanism actually produce the artifact the mitigation names (e.g., a real preservation theorem, not just the phrase "preservation theorem")? If keyword present but artifact absent → `mitigation_substantively_delivered = false`. There is no separate keyword-level validator (an earlier `V_anti_pattern` was deleted as redundant with this prompt-time check).

4. **paper_pointed_threat must search lit_table ∪ collision_hits**. Not just lit_table — collision_hits contains mechanism-specific recent retrievals that lit_table's broad-domain queries miss. If `threat_paper_id = no_threat_found` after honest search, that is a clearance signal — do NOT fabricate a generic threat to fill the field. "no_threat_found" is the correct answer when no paper-pointed threat exists.

5. **Verdict is two-layer: hard floor (mechanical) + soft judgment (LLM)**. Layer 1 is a corpus-anchored hard floor: triggered Reject pattern / unmitigated anti-pattern / exact-mechanism collision force `abandon`, period. The LLM cannot override these. Layer 2 is LLM judgment among the cases Layer 1 didn't reject — the LLM weighs how severe each borderline / partial finding is for advance vs revise. The LLM CAN promote borderline → advance when the borderline is in a non-load-bearing aspect (with concern surfaced for Phase 4 to fold into reviewer_concerns_and_responses). The LLM CAN demote clear → revise when holistic reading identifies a concern across-checks that no single check flagged. The LLM CANNOT rewrite per-check facts (a check that says borderline stays borderline). `verdict_rationale` must cite specific check findings (lesson_quoted / failure_mode_quoted / verdict label of named sub-block) — generic "all checks pass" or "looks fine overall" is a process error and not auditable.

6. **Why the two-layer split**: pure mechanical aggregation over-triggers revise (treats 1 trivial borderline same as 3 severe borderlines). Pure LLM verdict introduces agreeable-bias and loses audit trail. The hard floor preserves what's non-negotiable (corpus-anchored Reject facts); the soft layer lets the LLM use context to distinguish "must fix" from "noted concern, advance".
