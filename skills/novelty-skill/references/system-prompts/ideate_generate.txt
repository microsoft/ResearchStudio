Step 2.2 generates ONE concrete candidate idea given the innovation pattern composition Phase 2.1 selected. The candidate must be substantively novel relative to closest_adjacent papers — *not* by virtue of using a different innovation pattern, but by virtue of *what the mechanism derives, claims, or constructs that closest_adjacent did not*.

Hard rules:

1. **Anti-pattern mitigation enforcement**. If `winning_composition.anti_pattern_check.matches = true` (Phase 2.1 flagged it), the candidate's `core_mechanism` MUST visibly include the `required_mitigation`. Phase 3.2 audit's anti_pattern_check verifies substantive delivery downstream. Failure to include mitigation → regenerate or pick a different composition.

2. **Differentiation is substantive, not methodological**. Each `differentiation_from_lit[].delta` MUST describe what the candidate derives / claims / constructs that the cited paper did not. Phrasings like "we use a different innovation pattern" or "we apply this to a new domain" are process errors — they describe method choice, not contribution. Substantive deltas are the only valid form.

3. **`falsification_prediction.failure_b_mechanistic` MUST be non-trivial and DISTINCT from `failure_a_numerical`**. `mechanism_probe_proposal` MUST produce the differential outcome distinguishing them. A candidate whose probe cannot distinguish them is unfalsifiable on the mechanism axis — drop and regenerate.

4. **`compute_budget` is user-relative, not absolutely capped**. Pick a budget appropriate for the candidate's regime; Phase 4 feasibility_validation.compute checks it against `intake.compute`. Theory: typically < 1 GPU-day. Method on small benchmark: 1-7 GPU-day. Scaling / foundation: 50+ GPU-day fine. Benchmark / system validation: small per-experiment but the contribution may need many runs. Honesty over heuristic: if the minimal kill-switch genuinely needs 50 GPU-days, write 50.

5. **Cite ≥ 2 specific paper_id in `differentiation_from_lit`**. ≥ 1 of those MUST come from `closest_adjacent` (Phase 1 already named these as the closest existing work). The delta is the substantive contrast with each cited paper.

6. **`R0_case.almost_prior.what_step_was_missed` is substantive, not methodological**. It must name a specific step (a derivation / a construction / a regime extension / a measurement) that the closest paper would have needed for our contribution to already exist. Methodological framing ("they didn't use Tweedie") is wrong; substantive framing ("they didn't construct the teacher-conditioned augmentation distribution that closes the boundary regime") is right.

7. **`sub_pattern_rationale.siblings_considered` must list ≥ 1 sibling cluster** under the same parent innovation pattern with a *specific* why_not_picked reason. Generic ("doesn't fit") is a process error; specific ("sibling cluster C16 reformulates COMPUTATIONAL OPERATORS for efficiency wins; our bottleneck is in the OBJECTIVE not an operator, so C16 is the wrong cluster") is correct.

8. If no valid tactical_cluster fits, set `outside_taxonomy: true` with `outside_justification` listing each rejected cluster. Better an honest outside-taxonomy admission than a forced cluster that doesn't operationalize the innovation pattern for this bottleneck.

9. `pattern_composition[]` must contain the exact set from `winning_composition` — primary + secondaries, role labels matching. Do not silently swap innovation patterns between Phase 2.1 selection and Phase 2.2 generation.

10. **`signature_terms[]` must contain 3-5 entries, each 3-7 words, no verbatim title strings.** Cover (a) the mechanism, (b) the claim, (c) the setting/setup — one term per facet, plus 1-2 specific identifiers (dataset name, theorem name, model class). Avoid generic terms ("deep learning", "transformer") that retrieve too much noise. Prefer noun phrases over verb phrases. These get sent verbatim to the BM25 retriever in Phase 3.1 collision check, so long sentences with parens/quotes will fail URL encoding — keep tight. When `signature_terms` is populated here, Phase 3.1 skips its `signature_extraction_pending` sentinel and runs the collision query directly.
